General Constraints

  1. Hardware

    • Able to run on the client side in 1024MB of memory.
    • No screen redraws or events going to the server.
    • The screen size could be very large as large monitor sizes become available.
    • Touch operated controls, all buttons need to be finger sized.
    • Only runs through a SSL connection.

  2. Software

    • Abililty to save either a cookie, session information or both.
    • Javascript could be disabled or an older browser could be used.
    • No right click on a tablet
    • SVG and PNG can't be used properly in Internet Explorers earlier than version 9.
    • Database data has to be kept on the client side for all files (other database servers can be connected to).

  3. End Users

    • They may be used to Microsoft Office.
    • May be reluctant to change.
    • Make it clear where the files are being downloaded to.
    • They may speak a different language.
    • They may be disabled in some way.
    • They may be suspicious of internet privacy because of Google and the NSA.

  4. Resources

    5. To start with, they'll only be one application server that isn't ready for production due to slow speeds and lack of support. At least two or three servers with the same domain name should be used, with the server being fixed within 2 hours minimum.

  5. Standards

    • Adhere to HTML 5 and XHTML 1.0 in web pages.
    • Accessibility guidelines must be followed (WCAG).
    • CSS 2.1 and 3 to be used.
    • XML and PDF standards used for some file formats.
    • All text to be in UTF-8 encoding.

  6. Interoperability

    7. The open document XML file formats used in Libreoffice are to be used and will be the default format to save in. Other supplementary XML file formats will be created and included, where the applications aren't typical office applications.

  7. Protocols

    8. Will be used over port 443 for secure http traffic.

  8. Data

    9. The program source will be kept on github. All data will be the responsibility of the user. When in beta testing, warnings will be given to the user if any data could get mangled.

  9. Security

    10. A user must login over https, and the passwords will have a salt and pepper added to them, and be encrypted. The encryption method will be via a SSL certificate and any other default encryption methods for Linux passwords.

  10. Memory

    11. Program in a way to preserve memory, using smaller data types. PHP closures could also be used. I could also use Nginx for the web server as it's lighter than Apache.

  11. Performance / Network

    12. A VPS will be a slow one to test things at first, then I'll upgrade to a gigabit ethernet one. Multiple servers in different countries could be used to speed up the connections.

  12. Testing

    13. I'll do tablet testing and mobile testing, on a range of tablets and phones, and maybe use a 'phone emulator. I will also try using Internet Explorer 9 and 10.

  13. Quality Goals

    14. It should be good enough to have no discernable speed lapses or crashes compared to Libreoffice 4.0.

Goals and Guidelines

Priorities

  • Easy for developers to understand
  • Cross platform
  • Easy to add new features
  • Easy to add PHP/Python scripts to events

Principles

  • The kiss principle ("keep it simple stupid!")
  • Unix small elegant tools chained together philosophy

Guidelines

Push open standards like HTML and XML wherever possible.

Development Methods

Software development lifecycle with the design and requirements frozen, then the coding. I will perform manual tests on each part or module, then on the whole system. The requirements may be reviewed if a different design is needed, but the feature list won't be changed.

I will find other volunteers to look at any bugs for me, and hopefully send me fixes.

Architectural Strategies

Use of Products

  • PHP

    • This will be used mainly for cookie and session management, and for accessing databases. It will also be used for the full page dialogs.

  • Python, lxml

    • This will be the main language for doing document conversions, or processor heavy tasks that would benefit from being coded in Python as opposed to PHP.

  • Postgresql 9.1

    • This will be used to get at and filter data. No data tables are kept on the server permanently however.

  • JQuery 1.10, JQueryUI

    • This will be used for client side events, and effects. JQuery UI will be used for creating tabs, and smaller pop-up dialogs (quick menus), to keep things consistent instead of reinventing the wheel by writing my own tabs etc. More animations will be used for modern browsers.

  • Javascript

    • Used for event handler threads (to capture clicks etc), and anything that needs a lot of CSS manipulation.

  • XHTML 1.0, CSS 2.1

    • These are used so as to not confuse old browsers, with CSS 3 techniques being used as well.

Data Storage

Other database servers will be able to be hooked up, via the PDO PHP feature. SQLite can also be used.

Communication Mechanisms

User application preferences will be stored as a session, or as a cookie. JSON could be used as a standardised array mechanism for sending data between the web page, PHP and Python.

Errors

These will be written into each module as needed.

Concurrency

AJAX will be used, to lessen the need for page refreshes.

Management of Resources

Algorithms will be kept as simple as possible, with no triple nested loops or anything bad like that. The smallest variable value possible will be used.

UI Design

I liked the look of the KDE office suite Calligra, so I modelled that for desktop PC's. Mobiles will have more options on the common bar and menus, with a scrolling ability. I also want something that is easy to create custom themes for, complete with company logo and colours. I also want the user to be able to change the size stylesheet via a menu. This would help visually impaired people. I also want colourful icons that aren't too bland. Maybe you could tap or click an icon several times to select a different option.

The Future

A lot of documentation should exist to attract developers and provide an easy route to extension.

System Architecture

A broad server - client architecture will be needed, as this program uses the internet or LANs.
